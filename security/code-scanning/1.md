# Code scanning finding #1 — Use of a broken or weak cryptographic algorithm

**Finding:** [github.com/bitwise-media-group/setup-evolve/security/code-scanning/1](https://github.com/bitwise-media-group/setup-evolve/security/code-scanning/1)

| Field | Value |
| --- | --- |
| Rule | `js/weak-cryptographic-algorithm` (CWE-327, CWE-328) |
| Tool | CodeQL 2.25.6 |
| Severity | Warning (security-severity: high) |
| Location | `dist/index.js:25475` |
| Detected | 2026-06-13 against commit `c0db8f5` |
| Outcome | **False positive** |

## What CodeQL reported

The query flagged a SHA-1 hash and traced its input back to a value it labelled "sensitive data
from an access to `uid`":

```javascript
const digest = crypto.createHash('sha1').update(keyValue + uid).digest('base64');
```

## Why this is a false positive

**1. It is third-party code in a build artifact, not our source.**

`dist/index.js` is the bundled Action output produced by Rollup from `src/main.ts` (see
`rollup.config.js`). The flagged line lives inside [undici](https://github.com/nodejs/undici)
— Node's HTTP/WebSocket client — which Rollup inlines into the bundle. None of our own source
uses SHA-1: the only hashing in `src/` is SHA-256 for download integrity verification
(`src/download.ts`). Because the CodeQL workflow has no path filter, it analyses the committed
bundle and therefore scans every vendored dependency along with our code.

**2. The SHA-1 use is mandated by the WebSocket protocol, not a security choice.**

This is undici's WebSocket client validating the opening handshake defined by
[RFC 6455 §4.1](https://datatracker.ietf.org/doc/html/rfc6455#section-4.1). The server returns
`Sec-WebSocket-Accept = base64(SHA-1(Sec-WebSocket-Key + GUID))`, and the client recomputes the
same value to confirm the peer spoke the WebSocket handshake. SHA-1 here is a fixed,
protocol-defined checksum used to detect non-WebSocket endpoints — it provides no
confidentiality, integrity, or authentication guarantee, and the standard cannot be satisfied
with a different algorithm. Substituting SHA-256 would break every WebSocket connection.

**3. The "sensitive data" in the data-flow is a public constant.**

The value CodeQL calls `uid` is the WebSocket magic GUID
`258EAFA5-E914-47DA-95CA-C5AB0DC85B11` — a literal published in RFC 6455, not a secret or
user-controlled input. The taint-tracking premise behind this alert (weak crypto applied to
sensitive data) does not hold.

## Resolution

No code change is warranted: the construct is correct, lives in a vendored dependency, and is
required by the WebSocket specification. The action's own cryptography (SHA-256 for asset
verification) is unaffected. The finding is dismissed as a **false positive**.

To reduce recurring noise from scanning bundled dependency code, a future option is to add a
CodeQL configuration with `paths-ignore: [dist]` so analysis covers `src/` only — the bundle's
sources are already analysed directly.
